# Адаптеры

`LauncherAdapter` — центральный интерфейс для общения с бэкендом. Все операции с сервером проходят через него.

## Интерфейс

```dart
abstract interface class LauncherAdapter {
  Future<AuthToken> login(String login, String password);
  Future<void> logout();
  Future<List<Server>> fetchServers();
  Future<List<Profile>> fetchProfiles();
  Future<Profile> fetchProfile(String id);
  Stream<DownloadProgress> syncAssets(Profile profile, {Set<String> enabledMods});
  Future<List<String>> verifyIntegrity(Profile profile);
  LaunchArgs buildLaunchArgs(Profile profile, AuthToken token);
  Future<void> connect();
  Future<void> disconnect();
}
```

## GravitAdapter

WebSocket-адаптер для GravitLauncher.

| Аспект | Детали |
|--------|--------|
| Протокол | WebSocket (JSON-сообщения) |
| Пакет | `web_socket_channel` |
| Файлы | `core/adapter/gravit/gravit_adapter.dart`, `gravit_ws_client.dart` |
| Sync | SHA-1 хеширование, батчевое скачивание (4 потока), post-download верификация |
| Моды | Optional mods с enable/disable, зависимости между модами |

### Особенности sync

`syncAssets` запрашивает у сервера `hdir` (иерархический манифест файлов). Каждый файл имеет `digest` (SHA-1 base64), `size`, опциональный `url` и флаги.

Файлы классифицируются:
- **excluded** — скачиваются только если отсутствуют, хеш не проверяется (конфиги игрока)
- **flagged** — удаляются с диска (серверная команда на удаление)
- **optional mods** — скачиваются/удаляются в зависимости от выбора пользователя
- **обычные** — проверяются по хешу, скачиваются если не совпадает

После скачивания каждый файл проверяется повторно (post-download hash check). Если хеш не совпал — файл удаляется и помечается как failed.

### Path Sanitizer

Все пути из серверного `hdir` проходят через `PathSanitizer.isValid()`. Это защита от path traversal — сервер не может заставить клиент записать файл за пределы game directory.

<Callout type="warn" title="Безопасность">
PathSanitizer блокирует path traversal атаки — серверный манифест не может заставить клиент записать файл за пределы game directory.
</Callout>

## IntegrityStrategy

Перед запуском игры `LaunchGameUseCase` проверяет целостность файлов через `IntegrityStrategy`. Это слой безопасности, специфичный для каждого адаптера.

### Интерфейс

```dart
abstract interface class IntegrityStrategy {
  Future<IntegrityResult> verify(Profile profile);
}

sealed class IntegrityResult {
  const IntegrityResult();
}

final class IntegrityOk extends IntegrityResult {
  const IntegrityOk();
}

final class IntegrityFailed extends IntegrityResult {
  const IntegrityFailed(this.code, {this.detail});
  final String code;
  final String? detail;
}
```

Результат — либо `IntegrityOk` (файлы в порядке), либо `IntegrityFailed` с кодом ошибки (например, `'integrity.fileCorrupted'`).

### Стратегии по адаптерам

| Адаптер | Стратегия | Суть |
|---------|-----------|------|
| **Mock** | `NoopIntegrityStrategy` | Всегда возвращает `IntegrityOk` |
| **GravitAdapter** | `ForwardVerifyIntegrityStrategy` | Запрашивает свежий манифест с сервера через WS и сверяет с локальными файлами (SHA-1) |
| **CustomRestAdapter** | `SignedManifestIntegrityStrategy` | Загружает JSON-манифест с подписью Ed25519, проверяет подпись, потом сверяет файлы |

### ForwardVerifyIntegrityStrategy (GravitAdapter)

`LauncherAdapter` получает метод-геттер:

```dart
abstract interface class LauncherAdapter {
  IntegrityStrategy get integrityStrategy;
  // остальные методы...
}
```

Стратегия в начале запуска:
1. Отправляет WS-запрос `{'type': 'update', 'dirName': profile.gameDir}` 
2. Получает `hdir` (иерархический манифест файлов с SHA-1 в base64)
3. Сверяет каждый файл локально
4. Возвращает `IntegrityOk` или `IntegrityFailed('integrity.serverMismatch')`

### SignedManifestIntegrityStrategy (CustomRestAdapter)

**Конфиг адаптера расширен:**

```dart
class CustomRestConfig implements AdapterConfig {
  const CustomRestConfig({
    required this.endpoint,
    this.publicKeyHex = '',  // новое: публичный ключ Ed25519 в hex
    this.manifestEndpoint = '/manifest/signed',  // новое: где брать манифест
  });
  // ...
}
```

**Формат endpoint-а** (владельцы серверов):

```http
GET /manifest/signed?profileId=<profile_id>

Response:
{
  "manifest": "<json-string>",
  "signature": "<base64-encoded-ed25519-signature>"
}
```

**Структура манифеста** внутри `manifest`:

```json
{
  "version": 1,
  "files": {
    "relative/path/file.jar": "<sha1-base64>",
    "config.json": "<sha1-base64>"
  }
}
```

Стратегия:
1. Загружает `manifest` + `signature` от сервера
2. Верифицирует подпись Ed25519 над БАЙТАМИ строки `manifest`
3. Парсит JSON манифеста
4. Сверяет SHA-1 каждого файла
5. Возвращает результат

**Пример Node.js для генерации подписи** (владельцы серверов):

```javascript
const nacl = require('tweetnacl');
const crypto = require('crypto');

const manifest = JSON.stringify({
  version: 1,
  files: {
    'mods/mod.jar': 'hash1',
    'config.json': 'hash2'
  }
});

const manifestBytes = Buffer.from(manifest, 'utf8');
const secretKey = nacl.sign.keyPair.fromSecretKey(secretKeyBytes); // 64 bytes
const sig = nacl.sign.detached(manifestBytes, secretKey.secretKey);
const signatureBase64 = Buffer.from(sig).toString('base64');

response.json({
  manifest: manifest,
  signature: signatureBase64
});
```

### Ошибки целостности (l10n)

| Код | Ключ в AppLocalizations | Смысл |
|-----|------------------------|-------|
| `integrity.fetchFailed` | `integrityFetchFailed` | Не удалось скачать манифест |
| `integrity.signatureInvalid` | `integritySignatureInvalid` | Подпись не совпала или не распарсилась |
| `integrity.serverMismatch` | `integrityServerMismatch` | Файл на диске не совпадает с манифестом сервера (ForwardVerify) |
| `integrity.fileCorrupted(file)` | `integrityFileCorrupted({file})` | Файл скорруптирован или удалён (SignedManifest) |

`ErrorPresenter` маппит эти коды на локализованные строки, которые показываются пользователю.

## AdapterRegistry

Регистрация адаптеров централизована в `AdapterRegistry`. Это позволяет добавлять новые адаптеры без изменения кода, который их создаёт.

### Структура

```dart
class AdapterRegistry {
  void register(
    String type, {
    required AdapterFactory factory,
    required AdapterConfig Function(Map<String, dynamic>) fromJson,
  });

  LauncherAdapter create(AdapterConfig config);
  AdapterConfig parseConfig(Map<String, dynamic> json);
  List<String> get registeredTypes;
}
```

### Встроенные адаптеры

В `lib/core/adapter/adapter_registrations.dart`:

```dart
void registerBuiltinAdapters(AdapterRegistry registry) {
  registry.register(
    'gravit',
    factory: (config) => GravitAdapter(config as GravitConfig),
    fromJson: GravitConfig.fromJson,
  );
  registry.register(
    'custom_rest',
    factory: (config) => CustomRestAdapter(config as CustomRestConfig),
    fromJson: CustomRestConfig.fromJson,
  );
  registry.register(
    'mock',
    factory: (_) => MockAdapter(),
    fromJson: (_) => const MockConfig(),
  );
}
```

### Использование в DI

В `core/di/providers.dart`:

```dart
final adapterRegistryProvider = Provider<AdapterRegistry>((ref) {
  final registry = AdapterRegistry();
  registerBuiltinAdapters(registry);
  return registry;
});

final launcherAdapterProvider = Provider<LauncherAdapter>((ref) {
  final config = ref.watch(adapterConfigProvider);
  final registry = ref.watch(adapterRegistryProvider);
  return registry.create(config);
});
```

При загрузке конфига — парсим JSON через registry:

```dart
final adapterConfig = registry.parseConfig(configJson);
```

### Добавление своего адаптера

Три шага:

#### 1. Создай класс адаптера

```dart
// lib/core/adapter/my_adapter/my_adapter.dart
class MyAdapter implements LauncherAdapter {
  MyAdapter(this.config);
  final MyAdapterConfig config;

  @override
  Future<AuthToken> login(String login, String password) async {
    // Твоя реализация
  }
  
  // Остальные методы интерфейса...
}

// Конфиг
class MyAdapterConfig implements AdapterConfig {
  const MyAdapterConfig({required this.endpoint});
  final String endpoint;

  factory MyAdapterConfig.fromJson(Map<String, dynamic> json) {
    return MyAdapterConfig(endpoint: json['endpoint'] as String);
  }

  @override
  String get type => 'my_adapter';
}
```

#### 2. Зарегистрируй в `adapter_registrations.dart`

```dart
void registerBuiltinAdapters(AdapterRegistry registry) {
  // Существующие адаптеры...
  
  registry.register(
    'my_adapter',
    factory: (config) => MyAdapter(config as MyAdapterConfig),
    fromJson: MyAdapterConfig.fromJson,
  );
}
```

#### 3. Используй в конфиге приложения

```json
{
  "adapter": {
    "type": "my_adapter",
    "endpoint": "https://example.com/api"
  }
}
```

При загрузке конфига:

```dart
final json = jsonDecode(configString);
final config = registry.parseConfig(json['adapter']);  // Вернёт MyAdapterConfig
final adapter = registry.create(config);  // Вернёт MyAdapter
```

Registry сам разберётся какой класс использовать на основе `"type"`.

<Callout type="info" title="Новый адаптер = только 3 правки">
Чтобы добавить новый адаптер, нужно поменять только `adapter_registrations.dart`. Остальной код (DI, создание) не трогается.
</Callout>
